# PRK Question Flow v2.2 Upgrade Plan: Auto-QA Integration

**Date**: 2026-04-11  
**Purpose**: Automatically trigger QA validation after question generation  
**Version**: v2.1.0 → v2.2.0  
**Status**: DESIGN PHASE

---

## Executive Summary

Upgrade the prk-question-flow workflow to automatically invoke QA validation immediately after question generation, eliminating the manual step of running `prk-question-qa` separately.

**Key Change**: `prk-question-create` will automatically chain to `prk-question-qa` after generating questions.

---

## Current Workflow (v2.1)

### User Experience
```bash
# Step 1: Generate questions
prk-question-create 10
# ✓ Questions generated → saved to questions-{industry}.yaml
# ✓ Status: qa_status = TBI

# Step 2: User must manually run QA (MANUAL STEP)
prk-question-qa
# ✓ Questions validated
# ✓ Status: qa_status = PASSED or REVISED
```

### Problems
1. **Manual intervention required** - User must remember to run QA
2. **Easy to forget** - Questions can be exported without QA
3. **Inconsistent workflow** - Some users skip QA step
4. **Extra command** - Unnecessary friction in workflow

---

## Proposed Workflow (v2.2)

### User Experience
```bash
# Single command does both generation AND QA
prk-question-create 10
# ✓ Questions generated → saved to questions-{industry}.yaml
# ✓ Auto-QA triggered automatically
# ✓ Questions validated
# ✓ Status: qa_status = PASSED or REVISED
# ✓ Summary report shows both generation and QA results
```

### Benefits
1. ✅ **Zero manual intervention** - QA runs automatically
2. ✅ **Guaranteed validation** - All questions are QA'd before export
3. ✅ **Consistent workflow** - No way to skip QA
4. ✅ **Better UX** - Single command, less friction
5. ✅ **Faster iteration** - Immediate feedback on question quality

---

## Design Approach

### Option 1: Skill Chaining (RECOMMENDED)

**Concept**: `prk-question-create` skill automatically invokes `prk-question-qa` skill after generation.

**Implementation**:
- Modify `prk-question-create/SKILL.md` to include auto-QA step
- After generating questions, invoke the QA workflow
- Display combined results (generation + QA)

**Pros**:
- ✅ Clean skill-based architecture
- ✅ Maintains separation of concerns (agents remain independent)
- ✅ Easy to understand and maintain
- ✅ Backward compatible (can add `--skip-qa` flag if needed)

**Cons**:
- ⚠️ Slightly longer execution time (but user wanted this)

### Option 2: Unified Agent (NOT RECOMMENDED)

**Concept**: Create a single `question-generator-with-qa` agent that does both.

**Pros**:
- Single agent execution

**Cons**:
- ❌ Violates separation of concerns
- ❌ Harder to maintain
- ❌ Can't reuse QA agent independently
- ❌ Breaks existing architecture

### Option 3: Hooks (NOT APPLICABLE)

**Concept**: Use Claude Code hooks to trigger QA after file write.

**Cons**:
- ❌ Hooks are for shell commands, not skills
- ❌ Can't invoke Claude agents from hooks
- ❌ Wrong abstraction level

**Decision**: Use **Option 1: Skill Chaining**

```markdown
   - QA summary (Y PASSED, Z REVISED)
   - Next steps (revise if needed, or continue to next industry)
9. User must run command again for next industry
```

```
### Generation Commands

#### `prk-question-create [number]`
Generate questions for all industries and roles with automatic QA validation

**Parameters:**
- `number`: Questions per role (default: 10)
- `--skip-qa`: Skip automatic QA validation (optional)

**Behavior:**
- Reads current working folder from `state.yaml`
- Locks `industry.yaml` to prevent modifications
- Iterates through each industry in `industry.yaml`
- For each industry, iterates through all roles
- For each role, generates `number` questions with distribution:
  - **Pillars:** 2 questions per pillar (P, E, C, M, A)
  - **Difficulty:** 4 easy, 4 medium, 2 hard
- Saves to `questions-{industry_slug}.yaml` (one file per industry, all roles)
- Updates `create-status.yaml` after each industry
- **[NEW] Automatically runs QA validation:**
  - Validates format, content, quality
  - Semantic duplicate check via MCP (similarity > 0.85)
  - Scores: clarity, relevance, difficulty (1-10)
  - Sets qa_status: PASSED (≥7.0) or REVISED (<7.0)
  - Updates `qa-status.yaml`
- Displays combined generation + QA results
- Stops after each industry completion
- Auto-calls `prk-question-status` at end
```
